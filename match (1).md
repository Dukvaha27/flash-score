# Платформа текстовых трансляций спортивных событий "Матч"

Спортивный медиа-портал запускает платформу для текстовых онлайн-трансляций матчей. Комментаторы ведут текстовые репортажи в реальном времени: фиксируют голы, карточки, замены и другие события. Болельщики следят за ходом матча и обсуждают ключевые моменты в комментариях.

Мобильное приложение для болельщиков уже в разработке. Ваша задача - разработать бэкенд для управления матчами, ведения трансляций и взаимодействия с аудиторией.

## Важно

Прочитай задание до конца перед тем, как начинать писать код. Обрати внимание на бизнес-правила - особенно на типы событий матча, автоматическое обновление счёта при голе и формирование таймлайна.

## Архитектура

Проект должен следовать техническим требованиям буткемпа. Основные принципы:

- Микросервисная архитектура: каждый сервис - отдельный контейнер (допускается монорепозиторий)
- Каждый сервис - своя база данных (database per service)
- Слоистая архитектура внутри сервиса: transport -> services -> repository
- Go + Gin + GORM + PostgreSQL
- Kafka для асинхронного межсервисного взаимодействия
- HTTP для синхронных запросов между сервисами
- Docker и docker-compose для локальной разработки

## Сервисы

### Match Service (Дукхваха)

**Ответственность:** Матчи, команды, игроки, виды спорта, счёт.

#### Сущности

**Sport** - дисциплина, в которой проводится матч.

Примеры: football, basketball, mma, boxing, volleyball.

**Team** - спортивная команда.

Бизнес-правила:

- Команда привязана к виду спорта
- Имеет название, короткое название и город
- Может участвовать в нескольких матчах

**Player** - участник команды.

Бизнес-правила:

- Игрок привязан к команде
- Имеет имя, номер и позицию
- Используется для привязки к событиям матча (кто забил гол, кому карточка)

**Match** - спортивное событие между двумя командами.

Бизнес-правила:

- Матч привязан к виду спорта и двум командам (home/away)
- Создаётся в статусе scheduled с указанием даты и времени начала
- При старте переходит в live - фиксируется время начала
- При завершении переходит в finished - фиксируется время окончания
- Счёт обновляется автоматически по событию гола из Kafka-топика `match.goal`
- Отменить можно только scheduled матч
- Статусы: scheduled, live, finished, cancelled
- Дополнительные поля: место проведения (venue), название турнира (tournament_name)

#### Функционал

- CRUD матчей с фильтрацией по виду спорта, дате, статусу и пагинацией
- Управление статусами: начать, завершить, отменить матч
- Получение активных матчей
- Обновление счёта по событию гола из Kafka
- Список видов спорта
- CRUD команд с фильтрацией по виду спорта
- CRUD игроков в команде
- Матчи команды

#### Kafka (producer)

- `match.started` - матч начался (match_id, home_team, away_team, sport)
- `match.ended` - матч завершён (match_id, home_team, away_team, final_score)

#### Kafka (consumer)

- `match.goal` - при получении события гола Match Service обновляет счёт матча (match_id, team_id, new_score)

### Event Service (Турпал-Али)

**Ответственность:** События матча (голы, карточки, замены...), текстовые комментарии автора, пользовательские комментарии к событиям, реакции.

#### Сущности

**MatchEvent** - ключевой момент в ходе матча.

Бизнес-правила:

- Типы событий: goal, yellow_card, red_card, substitution, penalty, var_decision, half_start, half_end, full_time, injury, timeout
- Можно добавлять только в live матч (проверка через HTTP в Match Service)
- Фиксируется минута матча (minute)
- При типе goal - публикуется Kafka-событие `match.goal`, Match Service сам обновляет счёт
- Опциональная привязка к игроку (player_id) и команде (team_id)
- Описание события (text) - краткий комментарий: "Удар головой после подачи с углового"

**Commentary** - авторское описание хода матча.

Бизнес-правила:

- Текстовый репортаж от комментатора, отдельно от типизированных событий
- Отображается в таймлайне вместе с событиями
- Фиксируется минута матча
- Только один Commentary может быть закреплён (is_pinned) одновременно
- Можно добавлять только в live матч

**Comment** - сообщение болельщика под событием или комментарием автора.

Бизнес-правила:

- Привязан к событию (event_id) или к комментарию автора (commentary_id)
- Автор может редактировать свой комментарий

**Reaction** - эмоциональная реакция болельщика на событие.

Бизнес-правила:

- Типы: like, fire, shock, sad, laugh
- Реакция ставится на событие или комментарий автора
- Один пользователь - одна реакция на элемент
- Можно сменить тип или убрать реакцию

#### Функционал

- CRUD событий матча (голы, карточки, замены и т.д.)
- Полный таймлайн: события + комментарии автора, отсортированные по минутам
- CRUD текстовых комментариев автора с закреплением/откреплением
- Комментарии болельщиков к событиям и комментариям автора, с пагинацией
- Реакции на события (поставить, сменить, убрать, получить сгруппированные по типу)

#### HTTP (исходящие запросы)

- -> Match Service: `GET /matches/:id` - проверить, что матч в статусе live

#### Kafka (producer)

- `match.event.created` - новое событие (match_id, event_type, minute, description)
- `match.goal` - гол (match_id, team_id, player_id, minute, new_score)

### User Service (Сайд-Ахмад)

**Ответственность:** Пользователи, аутентификация.

#### Сущности

**User** - участник платформы.

Бизнес-правила:

- Авторизация через JWT (access + refresh token)
- Роли: fan (болельщик), commentator (комментатор матчей), admin
- Комментатор может добавлять события и текстовые комментарии к матчу
- Опциональное поле: любимый вид спорта (favorite_sport)

#### Функционал

- Регистрация и авторизация (JWT с access + refresh token)
- Получение и редактирование профиля
- Публичный профиль пользователя

---

### Notification Service

**Ответственность:** Уведомления для болельщиков о событиях матчей.

#### Сущности

**Subscription** - подписка болельщика на команду или вид спорта.

Бизнес-правила:

- Пользователь подписывается на команду (team_id) или вид спорта (sport_id)
- Одна подписка на сущность (уникальная пара user_id + entity)
- Можно отписаться в любой момент

**Notification** - уведомление для пользователя.

Бизнес-правила:

- Содержит: user_id, message, is_read, created_at
- Создаётся автоматически при получении Kafka-события для подписчиков команды/вида спорта
- `match.started` - "Матч начался: {home_team} vs {away_team}" для подписчиков команд-участников
- `match.ended` - "Матч завершён: {home_team} {score} {away_team}" для подписчиков команд-участников
- `match.event.created` (event_type=goal) - "Гол! {minute}' - {description}" для подписчиков команд-участников

#### Функционал

- Подписка/отписка от команды или вида спорта
- Список уведомлений пользователя с пагинацией
- Отметить уведомление как прочитанное
- Количество непрочитанных уведомлений

#### Kafka (consumer)

- `match.started` - создать уведомления для подписчиков команд-участников матча
- `match.ended` - создать уведомления для подписчиков команд-участников матча
- `match.event.created` - создать уведомления при event_type=goal для подписчиков команд-участников

### Gateway Service (Сайд-Ахмад)

**Ответственность:** API Gateway, маршрутизация запросов, JWT-валидация.

#### REST API (Gateway)

| Метод | Путь                  | Описание                              |
| ----- | --------------------- | ------------------------------------- |
| \*    | /api/auth/\*          | Проксирование на User Service         |
| \*    | /api/users/\*         | Проксирование на User Service         |
| \*    | /api/matches/\*       | Проксирование на Match Service        |
| \*    | /api/teams/\*         | Проксирование на Match Service        |
| \*    | /api/players/\*       | Проксирование на Match Service        |
| \*    | /api/sports/\*        | Проксирование на Match Service        |
| \*    | /api/events/\*        | Проксирование на Event Service        |
| \*    | /api/commentary/\*    | Проксирование на Event Service        |
| \*    | /api/comments/\*      | Проксирование на Event Service        |
| \*    | /api/reactions/\*     | Проксирование на Event Service        |
| \*    | /api/notifications/\* | Проксирование на Notification Service |
| \*    | /api/subscriptions/\* | Проксирование на Notification Service |

#### Дополнительно

- JWT-валидация на уровне Gateway
- Агрегация данных (например, матч + команды + текущий счёт + последние события в одном ответе)

## Git workflow

Допускается работа в монорепозитории - один репозиторий на всю команду со всеми сервисами.

1. Работай в feature-ветках: `feature/match-crud`, `feature/goal-event`
2. Делай коммиты с понятными сообщениями
3. Создай Pull Request в `main`
4. Попроси товарища по команде сделать код-ревью
5. После аппрува - merge в main
